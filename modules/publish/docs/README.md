## 1. Module Positioning

`publish` is the sixth executable module in the processing pipeline:

`ingest -> classify -> curate -> edit (when needed) -> translate -> publish -> site`

The module reads completed translated records (`translation_status = 'completed'`) from the database, packages them into a public-facing representation, writes publish metadata records back to canonical storage, and exports the static JSON, RSS, and stats files to the git-ignored public export directory (`data/publish_export/`).

The public-facing `site` presentation layer consumes only the outputs generated by this module.

### Core Architectural Separation (MVP Phase)
* **Downstream Only**: `publish` is a downstream-only consumer. It does not decide what is approved or rejected; it merely processes what has been marked as approved and successfully translated.
* **Separation from Presentation**: `publish` does not compile HTML, render CSS, or handle public routes. It only produces structured content feeds and indices (JSON/XML). The static `site` module is responsible for the actual presentation.
* **State Preservation**: Although the exported files are considered a "rebuildable layer," `publish` preserves a record of what has been published for each language (including publication times and URL slugs) in the canonical database to ensure stability and auditability.

---

## 2. Key Responsibilities

1. **Selection**: Query completed translated records from `translation_output` that are eligible for export.
2. **Transformation**:
   - Slugify the English title into a unique, URL-safe route shared across all languages of the same article.
   - Format publication timestamps.
   - Append standardized AI attribution/disclosure statements.
3. **Canonical Record Persistence**: Record export events in the `publish_record` and `publish_language_status` tables to track URL slugs per article and track language-specific publishing status.
4. **Static Export Compilation**: Generate language-specific indices (`<language>/index.json`), syndication feeds (`<language>/feed.xml`), and individual article files (`<language>/items/<slug>.json`), as well as global stats (`stats.json`).
5. **Rebuildability**: Provide a CLI command to completely clear the export folder and recreate all assets from database history.

## 4. Minimal CLI Usage

Validate the output directory and database connection:

```text
python -m modules.publish.src.cli validate
```

Run database migrations to create the `publish_record` and `publish_language_status` tables:

```text
python -m modules.publish.src.cli migrate --db-path data/canonical.db
```

Perform an incremental run (export only new items since last publish run):

```text
python -m modules.publish.src.cli run --db-path data/canonical.db --export-dir data/publish_export
```

Perform a full rebuild (clean the export directory and regenerate all files):

```text
python -m modules.publish.src.cli rebuild --db-path data/canonical.db --export-dir data/publish_export
```

Check publication stats:

```text
python -m modules.publish.src.cli status --db-path data/canonical.db
```
