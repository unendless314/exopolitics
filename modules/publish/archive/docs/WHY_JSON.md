# Why JSON over Markdown (Architectural Decisions)

**Status:** Active document  
**Updated:** 2026-06-17  
**Author:** Antigravity & User  

---

## 1. 為什麼選擇 JSON 而不是 Markdown？

我們將每一篇發布內容比喻為**「樂高積木」**，而 Markdown 是**「拼好的玩具」**。

### 1.1 資料結構化（樂高積木 vs 拼好的玩具）
每篇發布文章都包含多個獨立欄位：`display_title` (標題)、`summary_short` (簡短摘要)、`bullets` (3個重點) 及 `canonical_url` (原文連結)。
* **JSON (結構化積木)**：保持欄位獨立。網頁端 (`site` 模組) 可以自由拿取標題放在最上方、將重點做成漂亮的卡片，並把連結做成亮眼的按鈕。
* **Markdown (拼好的玩具)**：所有文字混在一起。網頁端如果只想在首頁卡片中呈現「摘要」而隱藏「重點」，就必須用程式去拆解 Markdown 字串，容易出錯且無彈性。

### 1.2 列表與彙整效率
* **JSON 模式**：發布模組會產生一個彙整好的 `index.json`，裡面收集所有文章的「標題」與「摘要」。網頁首頁只需讀取這一個小檔案，就能渲染出全部文章列表。
* **Markdown 模式**：若首頁想列出 100 篇文章的標題，網頁程式必須在伺服器端或瀏覽器端遍歷打開 100 個不同的 `.md` 檔案並逐一解析，效能極差。

### 1.3 資料與視覺呈現分離（彈性）
* **JSON 是純資料**：不帶有任何字體大小、顏色或排版樣式。若未來網站風格要大改版，只需修改 `site` 模組的樣式（CSS/HTML），發布層的 JSON 完全不用變更。

## 2. 為什麼不讓 `site` 模組直連資料庫，而需要 JSON 銜接？

### 2.1 靜態網站的運作本質（無資料庫）
靜態網站託管平台（如 GitHub Pages、Cloudflare Pages、Vercel）只提供靜態檔案下載，**沒有運行中的伺服器與資料庫**。
* 當讀者在網頁上進行動態搜尋、標籤篩選或下拉加載時，瀏覽器中的 JavaScript 可以直接請求 `index.json` 或 `items/<slug>.json` 來動態處理，不需要後端伺服器支援。

### 2.2 開發與部署隔離（安全防護）
我們的資料庫與爬蟲系統運行在私有的安全伺服器上，而網頁打包通常在第三方的雲端平台（如 GitHub Actions、Vercel Build Server）上進行。
* 透過 JSON 隔開，打包平台與讀者瀏覽器只需讀取已公開的 JSON 靜態檔案，**完全碰不到、也無需連線至私密且珍貴的 `canonical.db`**。

### 2.3 系統解耦（防止連帶損壞）
* 如果 `site` 直連資料庫，一旦未來我們修改資料庫欄位名稱（例如 `display_title` 改為 `title`）或更換資料庫系統（例如 SQLite 換成 PostgreSQL），網頁端程式就會直接崩潰。
* 透過 JSON 作為「API 契約」，只要 `publish` 輸出的 JSON 格式不變，後端資料庫做任何改動都不會影響前端網頁。

### 2.4 跨平台擴充性
JSON 是通用的資料格式。未來若需要開發 **手機 App (iOS/Android)**、**電子報訂閱系統** 或 **語音語意助理**，這些平台都可以直接消費相同的 JSON 發布檔，無需重新撰寫資料庫讀取程式。
