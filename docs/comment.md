# Architecture Notes：規劃調整備忘

**文件版本：** v0.2 草稿  
**更新日期：** 2026-05-27  
**狀態：** 備忘

---

## 1. 本輪規劃調整

這一輪討論後，架構方向做了三個重要調整：

- 從單體 pipeline 思維，改為明確的多模塊架構
- 從「Markdown 可能兼任主存儲」改為「DB 為 canonical source，發布層可重建」
- 從預留全域 `config/` 改為「配置先歸模塊所有，必要時再上提」
- 從「以分類決定抓取」改為「內容語義、抓取分片、抓取頻率三者分離」
- 從「只有聚合條目」改為「聚合流與編輯流並存，但必須保留來源追溯與 AI 揭露」

---

## 2. 為什麼這樣調整

### 2.1 模塊分離比單體流程更適合這個案子

因為這個系統天然就分成幾段責任：

- 抓取
- 分類
- 審核
- 發布
- 展示

如果現在不分開，後面會很難處理以下需求：

- 只重跑 LLM，不重抓 RSS
- 只重做發布層，不重做審核
- 只調整前端，不影響 DB
- 未來拆成獨立 repo 或獨立排程任務

### 2.2 先不要急著做全域配置

目前看得到的配置幾乎都屬於 `ingest`：

- sources
- categories
- fetch rules
- fetch groups
- schedule classes

這些不是全域配置。  
過早建立根目錄 `config/`，只會讓所有權變模糊。

### 2.3 根目錄 `docs/` 應該是總覽層

現在的根目錄文檔應負責：

- 說明產品與架構方向
- 定義模塊邊界
- 定義狀態與資料流

真正的實作細節，應放到各模塊自己的 `docs/`。

## 3. 後續建議

- 先建立 `modules/ingest/`
- 把現在的 feed config 視為 `ingest` 資產
- 讓 `category_id` 只負責語義歸檔
- 讓 `fetch_group` 與 `schedule_class` 成為獨立配置欄位
- 若未來引入改寫內容，應以 `edit_draft` 與 `published_piece` 分開建模
- `classify`、`review`、`publish`、`site` 先保留為規劃目標
- 等兩個以上模塊真的共享同一設定時，再考慮建立根層 `config/`
